## 2. What this app is (one paragraph)

Internal cockpit unifying the **smartlead-perf plugin** (performance) and
**email-infra plugin** (infrastructure) via the shared `sp_*` Supabase tables
both plugins maintain. The app **reads sp_\* exclusively** through additive
`vw_cockpit_*` views and **never calls Smartlead/InboxKit live** (Omar's
architecture mandate: "a static view of Supabase that updates daily"). It was
built by repurposing the legacy `web/` app from omnivate-ai-outbound (NOT
greenfield — direction pivoted 06-29 from the earlier "clean rebuild" plan).
Next.js 16 / React 19 / Tailwind 3 + Radix / Recharts / @supabase/ssr.

## 3. Data architecture (verified live 07-02)

Two ingestion paths feed sp_*:

1. **Daily sync** (smartlead-perf plugin, local `npm run sync`) — full pull
   each morning. `sp_sync_runs` logs it; last success 07-02 07:43 UTC.
   Populates `sp_daily_campaign_facts` (the performance **source of truth**,
   also feeds the 6 AM Slack summary), `sp_campaign_lifetime` (Smartlead's own
   cumulative analytics per campaign), `sp_mailboxes` mirror, facts tables.
2. **Live webhook capture** — Supabase Edge Functions (all ACTIVE, verify_jwt=false):
   - `smartlead-events` v9 → `sp_send_events` (intraday sends; 2,435 events on 07-02, latest 15:36)
   - `smartlead-category` v1 → `sp_lead_category_events` (Amzat's new category-change listener)
   - `email-infra-slack-approve` v2 → order-approval primitive (flips sp_decisions/sp_orders)

**Golden rule (from the 06-18 Slack under-count postmortem):**
`sp_daily_campaign_facts` / `sp_campaign_lifetime` are the SoT for stats;
`sp_send_events` rollups are a LIVE INDICATOR only. The UI respects this —
the "Today, live" strip is visually separate and labeled.

**Freshness semantics:** daily facts run ~1 day behind (as-of last business
day); live strip covers today. Verified vs Smartlead API directly on 07-02:
campaigns idle today matched EXACTLY (1,205/15/10/661); active campaigns
differed by exactly today's activity. Supabase = Smartlead as-of last sync.

## 5. App code — what changed vs. the legacy baseline

**Query layer rewritten (function names/signatures preserved):**
`lib/queries/{clients,analytics,campaigns,mailboxes,alerts}.ts` + root
`lib/queries.ts`. `lib/queries/pipelines.ts` untouched (pipeline_definitions/
pipeline_runs/prompt_library are REAL current tables). `client_analytics_config`
+ `app_settings` kept as APP-OWNED config (targets, thresholds, parent/child
hierarchy, burn threshold) — config, not pipeline data.

**Key semantics:**
- Active clients = `sp_clients.active=true` (currently: acceleration_partners,
  cylindo, omnivate, paycaptain). Sidebar labels via `clientLabel()` in
  lib/types.ts (AP/PayCaptain/Cylindo/Omnivate). `DEFAULT_CLIENTS` fallback
  trimmed to these 4 (a DB blip can't resurrect churned clients).
- Campaign table/cards = **LIFETIME** stats from sp_campaign_lifetime
  (the old model was cumulative; feeding daily facts caused Omar's "no data
  per campaign" bug). Sparklines get reconstructed cumulative series
  (baseline = lifetime − window sum, then running sum) so the component's
  delta computation works. Campaign drill-down panel charts stay DAILY facts.
- Alert model = sp_infra_alerts: status `open|resolved` (no "dismissed" —
  ack = resolve + note), severity `high|medium|low` (perf sp_alerts use
  critical/warning/info). All color maps/filters/counters handle BOTH
  vocabularies (`["critical","high"]` bucket etc.).
- Lifecycle vocabulary = sp_*: active/resting/reserve/warming/parked/burnt/
  retired/master (ramping/draining are dead concepts, kept in types for
  back-compat only).
- 999-day runway + ready_leads=0 are "not tracked in sp_*" sentinels —
  UI renders "Not tracked" (cards, runway widget; funnel starts at
  In Campaigns). NEVER render 999 as real.
- In-service capacity = mailboxes tagged `{client}_active` (tag NAMES via
  the view), excluding retired/parked/master. Planning divisor 25/day.

**Auth:** middleware (`lib/supabase/middleware.ts`) now REDIRECTS every
route except `/login` when no session (the legacy app had "no redirect — all
routes are public" — REQUIREMENTS.md wrongly claimed AUTH-1 built).
All dashboard routes `force-dynamic` (app/(dashboard)/layout.tsx) — no
build-time DB access, no baked data.

**Disabled behind `lib/flags.ts`** (routes return 410 + buttons disabled):
onboarding wizard (+nav entry hidden; pages redirect `/`), infra lifecycle
actions (rotate/drain/rest/order/master — their Trigger.dev tasks target the
retired mailbox_* model), campaign pause/resume (was live Smartlead, DEF-2),
legacy sync triggers. Trigger.dev whitelist trimmed to `run-pipeline`.
Re-enable path = Build 5 on sp_*-native primitives (e.g. reuse
`email-infra-slack-approve` for approvals).

**Repointed API routes:** analytics/{snapshots,history} · campaigns/[id]/detail ·
smartlead/campaign-stats (same response shape, sp_* sourced) · alerts/{recent,
[id]/resolve,[id]/acknowledge} (write sp_infra_alerts) · clients/[slug]/orders
(sp_orders) · clients/[slug]/mailbox-config (sp_clients) · mailboxes/[id]/timeline
(sp_actions_log) · tasks/recent-runs (now reports sp_sync_runs) ·
analytics/refresh → 501 until PORT-1.
